### ✨ 인터랙티브 애니메이션 프로토타입 스펙 (Tech Spec v5.1)

`isCritical: true`가 되는 순간, 다음 3가지 효과가 **동시적/순차적**으로 발생해야 합니다.

#### 1. 데이터 폭발 그래프 (The SEI Spike Graph)
*   **컴포넌트:** `<DataGraph Component>`
*   **위치:** 화면 중앙 하단 또는 좌측 섹션.
*   **기술 사양:** Canvas 또는 SVG 기반의 인터랙티브 라이브 그래프.
    *   **상태 변화 (State Logic):** `Stable` → `Rapid Spike`.
    *   **애니메이션 상세:**
        1.  (0s - 1s) 정상적인 추이로 부드럽게 데이터가 쌓임 (`Ease-out`).
        2.  (1s - 2s, **Critical Moment**) SEI 값이 급격히 수직 상승하며 그래프의 라인이 강한 빛의 잔상 효과(Afterglow/Bloom effect)와 함께 최고점을 찍습니다.
        3.  (2s 이후) 데이터 폭발과 함께 그래프 자체가 순간적으로 노이즈에 휩싸이며, 가장 높은 지점(`currentSEI`)만 빨간색으로 하이라이트 됩니다.
    *   **CSS/JS 요구사항:** `transition-duration: 0.3s;` (데이터 포인트마다), `animation: glitch-flash 0.15s infinite alternate;`

#### 2. 경고 오버레이 및 글리치 노이즈 (Glitch & Warning Overlay)
*   **컴포넌트:** `<Overlay Component>`
*   **위치:** 화면 전체 (`z-index: 999`).
*   **기술 사양:** CSS 필터(Filter)와 마스킹 기법을 사용합니다.
    *   **상태 변화 (State Logic):** `Normal` → `Glitch Mode`.
    *   **애니메이션 상세:**
        1.  **(글리치 노이즈, 0s)**: 화면 전체에 미세하고 빠르게 깜빡이는 수평선 패턴의 디지털 오류 효과(Chromatic Aberration/Scanline)를 적용합니다. (CSS `animation` 사용).

### 1. 상태 분류 (ENUM Type: `status`)
| Status | SEI 값 범위 | 의미 및 스토리텔링 연동 | 비주얼 연출 가이드라인 |
| :--- | :--- | :--- | :--- |
| **NORMAL** | $0 \le SEI < 25$ | 초기 단계. 사소한 비효율성 발생. '개인의 노력'으로 해결 가능하다고 착각하는 구간. (진지하고 무표정한 배경) | 안정적인 파란색/회색 계열의 그래프 라인. 낮은 노이즈 레벨. |

| **CRITICAL** | $SEI \ge 80$ | 임계점 돌파, '통제력 상실' 직전의 구조적 오류 증명 구간. (클라이맥스) | 빨간색 경고 영역 폭발. 글리치 노이즈(Glitch Noise), 오버레이 효과, 데이터 파동 시각화 필수. **`status: CRITICAL`** |

### 2. 임계점 정의 및 변화 공식
*   **T1 (경고 임계치):** SEI = 25.0

    *   **데이터 처리:** 이 지점을 통과하는 순간, `status`가 **WARNING**으로 강제 변경되어야 합니다.
*   **T2 (위기 임계치):** SEI = 80.0
    *   **의미:** 개인 역량만으로는 해결 불가능한 '구조적 시스템 오류'가 명확히 드러나는 지점.

// API Mocking Layer: 실제 서버 호출을 시뮬레이션하는 더미 서비스입니다.
/**
 * @description 외부 결제 게이트웨이(PayPal/Stripe) 연동을 모방합니다.
 * 이 함수는 비동기 처리, 로딩 시간, 성공/실패 상태를 정확하게 재현하여 
 * 프론트엔드 컴포넌트의 견고성을 테스트하는 것이 목표입니다.
 * @param amount 구매 금액 (USD)
 * @returns Promise<SuccessResponse | ErrorResponse>
 */
export const simulatePaymentApiCall = async (amount: number): Promise<{ success: boolean, message: string, error?: string }> => {
    console.log(`[API Mock] Attempting to process payment for $${amount}...`);

    // 1500ms의 지연 시간(Delay)을 주어 실제 네트워크 호출처럼 보이게 합니다.
    await new Promise(resolve => setTimeout(resolve, 1500));

    // 테스트 실패 케이스 시뮬레이션: 'FAIL'이 카드 번호에 포함되면 실패 처리
    const mockCardNumber = "XXXX-XXXX-XXXX-1234"; // 실제로는 사용자 입력받지만, 예시로 하드코딩합니다.
    if (mockCardNumber.includes("FAIL")) {
        return { 
            success: false, 
            message: "", 
            error: "결제 게이트웨이에서 승인 거부 오류(Authorization Failed)가 발생했습니다." 
        };
    }

    // 성공 케이스 시뮬레이션
    return { 
        success: true, 
        message: `${amount} USD 결제가 정상적으로 완료되었습니다. 진단 리포트를 확인해주세요.`,
        error: undefined
    };

3. **게이미피케이션을 넘어선 '챌린지 구조':** 단순 포인트를 얻는 게임화 단계를 넘어, 특정 임무나 과제를 성공적으로 완료할 때마다 지식 기반이 확장되는 '프로젝트형 학습' (Project-Based Learning)에 대한 니즈가 높습니다.
4. **커뮤니티 연계의 중요성:** 언어 학습을 '개인의 노력'으로 보지 않고, 같은 목표를 가진 사람들과 주기적으로 상호작용하고 결과를 공유하는 **'집단적 책임감(Social Accountability)'**이 동기 부여의 핵심 요소입니다.
5. **시간 절약형 콘텐츠 선호 (숏폼 최적화):** 학습 자체가 어려워서 긴 영상을 끝까지 보기 어렵습니다. 따라서 1분 이내에 하나의 명확한 문법 규칙이나 발음 포인트를 '충격적인 비교'를 통해 보여주는 방식이 가장 효과적입니다.

## 🎬 II. 중국어 숏폼 영상 자동화 기획 보고서 (프로토타입)

**주제:** 시스템 오류 시뮬레이션을 활용한 언어 학습 콘텐츠 (예: "당신이 이 문장을 틀린 이유, 시스템적으로 분석해 드립니다.")
**톤앤매너:** 전문적, 기술적 증명(Technical Proof), 충격 요법(Shock Value).

### A. 기획 콘셉트 및 핵심 구조화
*   **콘셉트:** **"언어 오류 진단기 (Language Error Diagnostic Simulator)"**
*   **핵심 전개 로직:** 시청자가 흔히 틀리는 문장/발음을 예시로 제시 → 시스템(영상)이 그 오류를 '구조적 문제'로 정의하고 분석 → 정확한 원리(기술적 증명)와 함께 올바른 대안을 제공하며 끝남.
*   **자동화의 핵심 가치:** **‘전문가 수준의 진단 과정’을 매번 재현하는 시스템 구축.**

### B. 자동화 워크플로우 (Automation Workflow) 및 모듈 정의
| 단계 | 프로세스/모듈 | 입력 데이터(Input) | 출력 산출물(Output) | 담당 에이전트 | 비고 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **1. 아이디어 발굴** | 트렌드 키워드 매칭 및 Pain Point 선정 | ① 중국어 학습 Pain Point 리스트 (위 보고서) ② 타겟 청중 관심사(베트남 커뮤니티 핫 토픽) | [주제]: 핵심 오류 문장/문법 포인트 세트 (5개 이상) | Researcher | 콘텐츠 주제 결정. |
| **2. 스크립트 작성** | 기술적 증명 기반의 스토리보드 구성 | ① 핵심 오류 문장 ② 오류에 대한 구조적 원리(API 정의 방식) ③ 쇼츠 시간 배분표(15초/30초) | [스크립트]: 오디오 대본 + 시각 연출 가이드 (Time-coded) | Writer | 설득력 확보가 최우선. |
| **3. 에셋 제작** | 애니메이션 및 비주얼 자료 생성 | ① 스크립트 기반의 Key Visuals ② 전문 데이터/통계 그래프(오류 발생 빈도 등) ③ 템플릿 Props (글리치 효과, 경고창) | [에셋]: 배경 영상 클립 + 인터랙티브 그래픽 파일 (.mov/.json) | Designer | 개발자 친화적 형태로 제작. |
| **4. 최종 통합 및 검증** | 미디어 포맷 최적화 및 배포 준비 | ① 스크립트 ② 에셋 ③ SEO 키워드 묶음 (롱테일) | [최종 산출물]: 완성된 쇼츠 영상 + Description Box 표준 포맷 | All Agent / Developer | 크로스 플랫폼(YT/IG) 분배. |

### C. 완료된 내용 요약
1.  **시장 분석 기반:** 베트남 청중의 니즈를 '체계성 부재'와 '실생활 연동 부족'이라는 시스템적 결함으로 정의하여, 콘텐츠 기획에 활용할 수 있는 구조화된 Pain Point 리스트를 확보했습니다.
2.  **자동화 로직 확립:** 단순한 영상 제작이 아닌, **[발굴] → [스크립트] → [에셋] → [통합]**의 4단계 모듈형 워크플로우를 정의하여, 모든 콘텐츠가 일관된 기술적 증명 구조(Systematic Proof)를 갖도록 했습니다.
