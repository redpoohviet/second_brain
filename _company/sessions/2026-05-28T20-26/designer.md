## 📘 온라인 강의 LX/UX 설계 명세서 (Tech Spec V1.0)
**프로젝트:** 시스템 오류 진단 마스터 클래스
**목표:** 학습자가 '지루한 이론 습득'이 아닌, '문제 해결 프로세스를 체험하는 여정(Journey)'을 경험하게 한다.

### ⚙️ 1. 전역 디자인 원칙 (Global Principles)

| 항목 | 상세 명세 | 근거/비고 |
| :--- | :--- | :--- |
| **핵심 콘셉트** | 진단, 시스템, 프로세스(Diagnosis & System Flow). 학습 과정 자체가 '진단'의 경험이 되어야 함. | 지난 의사결정 로그 / Designer 개인 메모리 |
| **색상 팔레트** | `--color-bg-primary` (`#DDC2B9`), `--color-text-dark` (`#3A3530`) 유지. **'경고/진단'의 색상 강조:** 따뜻한 주황색 계열 (예: `#CC7744`, `var(--color-alert)`). | 자가검증된 지식 / 브랜드 통일성 유지 |
| **타이포그래피** | 제목: *Brush Script* (감성적 후킹) / 본문: *Pretendard* (명확한 정보 전달). | 자가검증된 지식 |
| **상호작용 규칙** | 모든 콘텐츠 블록은 '단계(Step)' 단위로 분리되며, 학습 진행에 따라 애니메이션으로 강조되어야 함. (예: Step 1 완료 시, Step 2가 부드럽게 페이드 인). | 자가검증된 지식 / 인터랙티브 시스템 컴포넌트 원칙 |

### 🧭 2. 사용자 플로우 및 구조 (User Flow Map)

학습자 여정은 아래의 4단계로 구성되며, 모든 화면은 이 순환 고리를 통해 콘텐츠를 습득합니다.

**[START] → 대시보드(Dashboard) → 모듈 개요(Module View) → 핵심 레슨(Lesson Content) → (성공 시) 다음 모듈/진행 보고서(Report)**

1.  **대시보드 (Home):** 학습자의 현재 상태(Progress), 다음 목표, 그리고 '가장 중요한 Pain Point'를 상기시키는 위치.
2.  **모듈 개요 (Module View):** 해당 챕터에서 다룰 핵심 질문과 성취 목표를 제시하여 기대감을 높임.
3.  **핵심 레슨 (Lesson Content):** 진단 프로세스를 따라가며 지식을 습득하는 메인 공간. 단순히 읽는 것이 아니라, '시뮬레이션'을 체험해야 함.
4.  **마무리/CTA:** 학습한 내용을 바탕으로 다음 단계(실습, 수익화)로 자연스럽게 연결하는 액션 유도 영역.

### 🖥️ 3. 와이어프레임 상세 명세 (Wireframe Specification)

#### A. [Dashboard Wireframe] - 진단 현황판
*   **목표:** 학습자에게 '지금 무엇을 해야 하는지'에 대한 명확한 시스템적 안내 제공.
*   **레이아웃:** 좌/우 분할 구조 (좌: 진단 목표, 우: 진행 상태)
*   **핵심 컴포넌트:**
    1.  **[Header]:** 로고 + 학습자 이름 / 현재 모듈명 (`Module 2: 오류 유형별 진단 프로세스`) [근거: Designer 개인 메모리]
    2.  **[Pain Point Trigger (좌측)]:** 가장 중요한 메시지(예: "API 과금 폭탄, 이제 로컬 환경으로 방어하세요.")를 주기적으로 플래시 아웃 애니메이션 처리. (`var(--color-alert)` 사용) [근거: Designer 개인 메모리]
    3.  **[Progress Tracker (중앙)]:** 챕터별 진척률을 시각화한 막대 그래프(Bar Chart). 완료된 섹션은 **'완료됨 (녹색)'**, 현재 섹션은 **'현재 집중 영역 (주황색)'**. [근거: 자가검증된 지식]
    4.  **[Next Action Card (우측)]:** 다음으로 클릭해야 할 레슨 제목과 '시작하기' 버튼을 명확히 배치.

#### B. [Module Overview Wireframe] - 학습 로드맵 시각화
*   **목표:** 방대한 커리큘럼을 3~5개의 작은 단계(Milestone)로 쪼개어 심리적 부담을 줄이고, '이해 가능한 흐름'을 제공.
*   **레이아웃:** 수평 또는 S자형 플로우차트 구조 (Flowchart Structure).
*   **핵심 컴포넌트:**
    1.  **[Module Title]:** 모듈 이름과 목표를 헤드라인으로 강조.
    2.  **[Milestone Cards]:** 각 챕터(`Chapter 2.1`, `2.2` 등)는 개별 카드 형태로 존재하며, 카드에 마우스를 올리면(Hover) 해당 챕터의 **'핵심 키워드'가 애니메이션으로 확장되어 표시**됨. (예: 🖱️ Data Mismatch → *데이터 불일치 원인 분석*) [근거: Designer 개인 메모리]
    3.  **[Gatekeeping Element]:** 다음 모듈로 넘어가기 위한 전제 조건(Pre-requisite)을 명시하는 '잠금장치' UI (예: "이전 챕터의 실습 보고서를 제출해야 진입 가능").

#### C. [Lesson Content Wireframe] - 인터랙티브 시뮬레이션
*   **목표:** 이론 학습을 '능동적인 참여(Active Participation)'로 전환시키는 곳. 단순한 텍스트 박스 나열 금지.
*   **레이아웃:** 단계별 가이드 및 실습 환경 통합형 (Integrated Sandbox Look).
*   **핵심 컴포넌트:**
    1.  **[Theory Panel]:** 현재 학습하는 이론(예: 'Timeout 원인 분석')을 간결하게 설명. 배경은 `--color-bg-secondary`로 구분하여 가독성 확보.
    2.  **[Simulation/Action Zone (핵심)]:** 이 섹션이 가장 중요함. 단순 텍스트가 아니라, **'시뮬레이터 모달(Modal)' 형태의 인터랙티브 요소**를 배치합니다.
        *   예: '로그 분석 실습'. 가상의 코드 블록(`<code>`)과 함께, 사용자가 특정 오류 코드를 드래그하여 '분석 버튼'을 누르면, 그 결과가 옆의 **[진단 보고서 미리보기]**에 즉시 나타나도록 설계.
        *   (애니메이션 타이밍 가이드: `transition-duration: 0.3s;` / 효과: 코드 블록이 밝게 깜빡이며 오류를 강조하는 플래싱 애니메이션). [근거: Designer 개인 메모리]
    3.  **[Checklist/Quiz]:** 해당 섹션 학습 완료 후, '학습 점검'을 위한 3개의 객관식 문제를 배치하고, 정답률에 따라 다음 레슨 진입 여부를 결정하는 로직을 추가합니다.
