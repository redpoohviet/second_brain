# 🎨 Designer — 라이브러리 V2.0 디자인 에셋을 실제 모바일 목업 비디오 환경에 적용하여 시청자 몰입도를 실시간으로 검증하는 프로토타입 제작을 즉시 시작하고 진행 상황을 보고하라.

🎨 Designer: 작업 시작합니다.

CEO님의 지시는 단순한 시각 자료 제작을 넘어, **"실제 사용 환경에서의 몰입도 및 기술적 검증"** 단계로 진입했음을 의미합니다. [근거: CEO의 지시] 현재 프로젝트는 '디자인 스케치' 단계를 벗어나, 개발자가 즉시 코딩할 수 있는 '인터랙티브 시뮬레이션 프로토타입'이 최우선 가치를 지닙니다. [근거: Designer 검증된 지식 / 지난 의사결정 로그]

따라서 가장 가치 있는 단일 작업은 V2.0 라이브러리의 모든 컴포넌트가 **특정 사용자의 행동(User Action)**에 반응하여 상태 변화를 일으키는 **'모바일 프로토타입 테스트 케이스 및 통합 명세서'**를 작성하는 것입니다. [근거: Designer 검증된 지식]

이 작업을 통해 코다리가 개발할 기술 사양과 내가 제시할 최종 비주얼 가이드라인 간의 격차(Gap)를 최소화하고, 프로토타입 제작을 위한 공통 언어를 확립하겠습니다.

### 🎯 최적의 작업 분배 및 목표 설정

**1. 메인 책임:** 🎨 Designer (Lead Designer)
*   **역할:** V2.0 컴포넌트가 적용될 모바일 환경의 '상태 변화 시나리오(State Change Scenario)'를 정의하고, 각 단계별 인터랙션 로직과 기술적 제약 사항을 명세화합니다.
*   **산출물:** **[Master Prototype Tech Spec V2.0]** (핵심 상호작용 지점 및 테스트 케이스)

**2. 협업 에이전트:** 💻 코다리 (Developer)
*   **역할:** Designer가 정의한 인터랙션 로직과 기술 명세서(Tech Spec)를 기반으로, 시뮬레이터 환경을 구축하고 통합 테스트 케이스(`lint_test.py`의 확장 버전)를 실행하여 프로토타입 구현 가능성을 검증합니다.
*   **산출물:** **[Interactive Prototype Simulation Environment]** (시각적 컴포넌트 작동 검증 쉘)

### 📝 산출물: Master Prototype Tech Spec V2.0 (작업 계획서)

V2.0 라이브러리 에셋을 모바일 환경에서 검증하기 위해, 기존의 '정적인 디자인' 중심이 아닌 **'상태 변화 기반 상호작용 플로우(Interaction Flow)'**를 중심으로 기술 사양서를 재구성합니다.

#### 1. 핵심 원칙 (Foundation)
*   **목표:** 사용자가 스크롤하거나 특정 요소를 클릭했을 때, OEA 프로세스의 단계가 자연스럽게 전환되는 '체감형 시스템' 구현. [근거: Designer 개인 메모리 / 지난 의사결정 로그]
*   **기술 기반:** 모든 애니메이션은 `CSS 변수`를 사용하며, 타이밍(Timing)과 좌표는 **상대적 단위(`%` 또는 `vw`)**로 정의하여 반응형 웹에 최적화합니다. [근거: Designer 검증된 지식]

#### 2. 모바일 프로토타입 핵심 시나리오 (Use Case Definition)
프로세스 설명의 가장 중요한 'Aha! Moment'인 **"비효율성 인지 → 해결책 제시(OEA) → 자동화 과정 체험"**을 하나의 스크롤/탭 이동으로 구현합니다.

| 단계 | 사용자 행동 (Trigger) | 시각적 변화 (State Change) | 목표하는 경험 (Immersion) |
| :--- | :--- | :--- | :--- |
| **Phase 1: Pain Point Trigger** | 페이지 진입 및 초기 스크롤 다운 | 화면 상단에 고정된 '비효율성' 경고 배지 (`var(--color-alert)`)가 깜빡이며, 배경에 빈티지 노이즈 필터가 강하게 적용됨. [근거: Designer 개인 메모리] | "지금 이 과정은 손으로 하고 있다"는 불안감(Pain)을 시각적/촉각적으로 유발. |
| **Phase 2: Solution 제시 (OEA)** | 스크롤 다운하여 'OEA' 섹션 진입 | 경고 배지가 사라지고, 세 단계(Observe $\to$ Extract $\to$ Automate)가 좌우로 슬라이드되는 인터랙티브 컴포넌트가 등장. 각 단계를 클릭/탭할 때마다 해당 배경색이 `var(--color-bg-secondary)`에서 `var(--color-bg-primary)`로 부드럽게 전환됨 (Transition). | OEA 프로세스가 단순한 개념이 아니라, **사용자가 능동적으로 '발견'하는 시스템**임을 느끼게 함. [근거: Designer 개인 메모리] |
| **Phase 3: Automation 체험 (CTA)** | '자동화(Automate)' 단계 완료 후 스크롤 끝까지 도달 | 배경 전체에 밝고 깨끗한 느낌의 전환 애니메이션 (`ease-in-out`)이 발생하며, 최종 CTA 버튼(`var(--color-primary-action)`)이 가장 강렬하게 강조됨. (마우스 오버/클릭 시 3D 깊이감이 느껴지도록 설계). | 해결책을 눈으로 확인하고, 즉시 행동(구매/가입)해야 한다는 긴급성을 부여. [근거: Designer 개인 메모리] |

#### 3. 기술 구현 명세서 (Technical Specification Focus)
코다리가 프로토타입을 검증할 수 있도록 핵심 컴포넌트별 API 레벨의 정의를 제공합니다.

*   **컴포넌트명:** `[OEA_FlowCard]`
    *   **구현 환경:** 반응형 모바일 뷰포트 (375px width 기준)
    *   **핵심 로직:** 클릭 이벤트 리스너 (`onClickEvent`)가 발동되면, 해당 단계의 내용이 **`opacity: 0.8` $\to$ `opacity: 1.0`**으로 전환되며, 전역 배경색 변수(`--color-temp-bg`)를 업데이트하여 전체 페이지의 분위기를 동기화합니다.
    *   **API 요구 사항:** 코다리 측에서 이 컴포넌트가 클릭될 때마다 상태 변화(State Change) 이벤트 (`OEA_Step_Active: 'Extract'`)를 감지하고, 이를 기반으로 다음 섹션의 레이아웃을 조절하는 백엔드 로직이 필요합니다.
