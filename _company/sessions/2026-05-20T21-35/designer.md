### ⚙️ 에셋별 기술 사양 상세 명세 (Tech Spec Detail)

#### **1. 구조적 오류 지표 (Structural Error Index)**
*   **목표:** 시청자에게 "뭔가 잘못되고 있다"는 즉각적인 불안감(Pain Point)을 주입한다.
*   **크기/배치:** 모바일 목업 기준, 9:16 비율의 좌측 상단 코너 (250px x 180px 권장).
*   **포맷:** SVG (확대/축소에 무한 확장 가능해야 하므로).
*   **애니메이션 요구사항:**
    *   **베이스 애니메이션:** 배경 다이얼은 느리게 회전하는 아날로그적 노이즈(Noise) 패턴을 유지한다. (`animation: spin 30s linear infinite;`)
    *   **오류 지표 움직임:** 핵심 포인터 바늘(Needle)의 색상(`var(--color-error-red)`)과 떨림 폭(Jitter)이 **시간 경과에 따라 급격하게 증가**해야 한다.
    *   **트리거 이벤트:** 특정 스크립트 지점(Pain Point 언급 시)에서 바늘이 갑자기 최대치로 '튕겨 오르는' (Snapping/Spiking) 애니메이션을 구현하고, 이와 동시에 경고음과 함께 붉은색 노이즈 레이어가 배경에 강하게 투사되어야 한다.

#### **2. 시간 경과 시뮬레이션 그래프 (Temporal Drift Graph)**
*   **목표:** 개인의 노력 부족 문제가 아닌, 시스템 자체가 변하는 '구조적 문제'임을 데이터로 증명한다.
*   **크기/배치:** 모바일 목업 기준, 중앙 하단 메인 정보 영역에 가로 폭 전체 사용 (360px x 120px).
*   **포맷:** SVG 또는 Canvas API (정확한 좌표와 실시간 데이터 처리가 필요하므로).
*   **애니메이션 요구사항:**
    *   **라인 트래킹:** 세 가지 가상의 변수(예: '개인 노력', '시장 흐름', '시스템 효율')를 나타내는 3개의 라인이 그려지며 시작한다. (Ease-in/Out 곡선 사용).
    *   **드리프트 시퀀스:** 영상 초반에는 3개 선이 평행하거나 안정적인 패턴을 유지하다가, 시스템 오류 지점(시간 $T_{error}$)에 도달하는 순간부터 **두 개의 라인(시장 흐름 vs. 개인 노력) 간의 거리가 급격히 벌어지며 이탈**해야 한다.
    *   **시각적 표현:** 가장 많이 이탈한 구간을 배경색이 어두워지는 '그림자 효과'와 함께 하이라이트 처리하여 시청자의 주의를 집중시킨다.

#### **3. 데이터 추출 시각화 컴포넌트 (Data Extraction Visualizer)**
*   **목표:** OEA 프로세스 중 'Extract(추출)' 단계를 기술적으로 구현하고, 지식의 가치를 높인다.
*   **크기/배치:** 모바일 목업 기준, 9:16 비율의 중간 과정 설명 시퀀스 내부에 배치 (300px x 250px).
*   **포맷:** SVG + Canvas API (유기적인 선과 점을 그리기 위해).
*   **애니메이션 요구사항:**
    *   **노드 생성:** 좌측에서 '혼란스러운 데이터 소스(Messy Data Source)'를 상징하는 무작위 노드들이 무질서하게 팝업된다.
    *   **연결 및 패턴 인식 (핵심):** 사용자가 특정 개념을 언급하거나, 시스템이 분석을 시작할 때 트리거되어, 이 혼란스러운 노드들 사이에서 **가장 논리적으로 연결되는 '황금 경로(Golden Path)'만을 찾아내는 빛나는 선(`var(--color-success)`)**이 애니메이션으로 그려져야 한다.
    *   **피니시:** 추출된 데이터는 깔끔하게 정돈된 박스 형태로 쌓이며, 마치 컴퓨터가 데이터를 분류하듯 미세한 '픽셀화(Pixelation)' 효과를 거쳐 안정화된다.

#### **4. OEA 프로세스 플로우 컴포넌트 (OEA Flow Component)**
*   **목표:** 복잡한 방법론을 구조적이고 명확하게 제시하며, 시청자에게 해결책에 대한 '확신'을 준다.
*   **크기/배치:** 모바일 목업 기준, 9:16 비율의 메인 솔루션 설명 영역 (320px x 150px).
*   **포맷:** CSS 변수 기반 컴포넌트 아키텍처 (개발 용이성을 최우선으로 고려).
*   **애니메이션 요구사항:**
    *   **기본 상태:** 세 개의 블록(Observe, Extract, Automate)은 수평 배열되며, 각각 다른 배경색(`var(--color-bg-secondary)` 등)과 아이콘을 가진다.
    *   **활성화/전환 애니메이션:** 시청자가 'OEA 프로세스의 중요성'에 주목하는 순간 (예: 스크립트가 OEA를 언급할 때), **현재 활성화된 단계의 블록만 부드럽게 커지면서(Scale Up) 빛나고, 나머지 단계는 은은한 배경으로 후퇴**해야 한다.
    *   **상호작용:** 개발자는 각 블록에 `:focus` 상태와 `data-active="true"`를 이용한 CSS 애니메이션을 적용하여, 단순한 정보 나열이 아닌 '단계별 진입'의 느낌을 주어야 한다.
